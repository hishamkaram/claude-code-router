# CCR v0.4.0 Acceptance Criteria

This is the release source of truth for local Claude subscription account
pools. v0.4.0 is the next version after the released v0.3.0 acceptance record.
A checked item requires reproduced evidence. Skipped live tests and unsupported
commands are not evidence.

## Command Surface

- [x] `ccr claude-account import <name> --from current` imports current-login
  credentials on supported platforms.
- [x] `ccr claude-account import <name> --oauth-token-stdin` stores an OAuth
  access token from stdin.
- [x] `ccr claude-account list`, `show <name>`, and `test <name>` expose only
  redacted local metadata and never print token values.
- [x] `ccr claude-account list --json` and `show <name> --json` emit stable,
  schema-versioned redacted output.
- [x] `ccr claude-account refresh <name> --from current` and
  `ccr claude-account refresh <name> --oauth-token-stdin` explicitly replace
  stored credentials without calling an undocumented refresh endpoint.
- [x] `ccr claude-account enable <name>`, `disable <name>`, and
  `remove <name> --yes` update account state and keychain cleanup behavior as
  documented.
- [x] Missing names, conflicting positional/`--name` values, unknown credential
  sources, and missing `--yes` fail before side effects.

## Subscription Pool Routing

- [x] `ccr launch --auth-mode subscription-pool` atomically selects and stamps
  the least recently used enabled, unexpired, non-cooling account without
  claiming an exclusive process-lifetime lease.
- [x] `ccr launch --auth-mode subscription-pool --claude-account <name>` selects
  only the named account and never rotates to another account.
- [x] `--claude-account <name>` without `--auth-mode subscription-pool` fails
  before launch side effects.
- [x] The selected account is visible in launch stderr and `ccr status`, while
  OAuth token values remain redacted.
- [x] Account identity is fixed for the Claude Code process; CCR does not
  perform an in-process identity swap.
- [x] If no usable account exists, the launch fails visibly and never falls
  back to the default Claude login, Anthropic API key, or another provider.

## Exhaustion and Relaunch

- [x] First-party Anthropic HTTP 429 responses during a pool launch mark the
  selected account cooling down with bounded retry timing.
- [x] Automatic relaunch happens only for plain interactive
  `ccr launch --auth-mode subscription-pool`: no `--print`, no
  `--claude-account`, no managed CUA options, and no extra Claude Code
  arguments.
- [x] Automatic relaunch stops the exhausted Claude Code process and starts the
  next attempt with `--continue`.
- [x] Explicit-account, print-mode, and argument-bearing launches return a
  visible rate-limit error instead of automatically relaunching.
- [x] Registered Anthropic-compatible provider 429 responses and
  OpenAI-compatible provider 429 responses do not trigger subscription-pool
  cooldown or relaunch handling.

## Secrets and Local Use

- [x] SQLite stores account metadata and keyring references only, never raw
  OAuth access or refresh tokens.
- [x] OAuth tokens are stored only in the OS keychain; keychain unavailability
  prevents account import before SQLite persistence.
- [x] macOS current-login import is rejected with guidance to use
  `claude setup-token` and `--oauth-token-stdin`.
- [x] Documentation presents subscription pools as local individual use and
  recommends official Anthropic API authentication for teams, automation,
  hosted products, and third-party integrations.
