# CCR v0.2.2 Acceptance Criteria

This is the release source of truth for image input on OpenAI-compatible
(LiteLLM) routes. A skipped live test or an unsupported command is not evidence.
Gates that were not executed in this release environment are left unchecked and
named explicitly; they are not claimed as passed.

## Image Input on OpenAI-Compatible Routes

- [x] Vision follows the discovered or overridden capability on the
  OpenAI-compatible path: reported/overridden `true` is forwarded, explicit
  `false` still returns HTTP 501, and unknown is forwarded for the provider to
  decide. The route no longer force-masks vision to false.
- [x] PDF and audio input remain masked to false on the OpenAI-compatible path,
  because the adapter cannot serialize document or audio content.
- [x] Anthropic `image` blocks translate to OpenAI `image_url` content parts
  (base64 becomes an inline `data:` URL; `url` sources pass through). Text-only
  requests keep the plain-string content form and are byte-for-byte unchanged.
- [x] `tool_result` images surface in a trailing user message. Every tool
  message for a turn is emitted first, then a single user message, so tool
  responses stay contiguous after the assistant `tool_calls` message even on
  parallel tool calls.
- [x] Empty text parts are dropped from multipart content, and an image-only
  `tool_result` uses a non-empty placeholder for backends that reject empty tool
  content.

## Review

- [x] A multi-agent review (four dimensions plus adversarial verification) ran
  against the uncommitted change. It surfaced a tool-message contiguity defect
  (a user message interleaved between tool messages on parallel tool results),
  which was fixed with regression tests before merge.
- [ ] The required uncommitted Codex review was not run in this environment.
