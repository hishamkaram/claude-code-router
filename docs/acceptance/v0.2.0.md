# CCR v0.2.0 Acceptance Criteria

This document is the release source of truth for runtime visibility,
conformance, lifecycle tracking, diagnostics, compatibility canaries, and team
profiles. A checked item requires reproducible evidence in the release pull
request. Skipped live tests do not count as evidence.

## Runtime Visibility

- [x] `ccr launch` creates a launch record before the gateway accepts traffic
  and finalizes it on every startup, cancellation, and process-exit path.
- [x] Authenticated message and token-count requests record the actual route,
  provider model, protocol, request capabilities, result, latency, and observed
  token usage without storing request or response content.
- [x] `/model` changes update the active route on the next request and are
  visible through `ccr status`, `ccr trace`, and the CCR status line.
- [x] Unknown CCR model IDs fail visibly and never fall through to Anthropic or
  another configured provider.
- [x] Trace persistence failures do not break routing and are visible as an
  observability degradation.
- [x] Event retention is bounded to 30 days and 10,000 combined events, and
  `ccr trace purge` removes history safely.

## Claude Lifecycle

- [x] One CCR launch can contain multiple Claude sessions created by startup,
  resume, and clear lifecycle events.
- [x] Subagent, task, teammate-idle, stop-failure, and session-end hooks update
  normalized session, agent, and task state.
- [x] Abrupt process exit marks unfinished sessions, agents, and tasks
  abandoned.
- [x] Existing Claude hooks are preserved. A managed-policy block is reported
  as unobserved rather than silently represented as an empty lifecycle.
- [x] CCR injects a launch-only status line only when no status line is already
  configured and never writes a Claude settings file or global model cache.

## Privacy And Security

- [x] SQLite, logs, CLI output, JSON output, test artifacts, and exported
  profiles contain no provider secret value, authorization header, prompt,
  response, tool argument, task description, transcript path, or raw hook body.
- [x] Lifecycle and status endpoints are loopback-only, use a separate
  ephemeral token, compare credentials in constant time, enforce methods and
  content types, and limit hook bodies to 64 KiB.
- [x] `--no-history`, `--no-lifecycle`, and `--no-statusline` independently
  disable their launch-scoped behavior.
- [x] Monetary cost is not estimated; traces distinguish observed token usage
  from unavailable usage.

## Conformance And Diagnostics

- [x] Provider conformance executes configuration, discovery where supported,
  text, stream, forced-tool, thinking, token-count, cancellation, and sanitized
  error checks through the production gateway path.
- [x] Capability-disabled checks are `not_applicable`; failure of a declared
  capability fails the run and never changes compatibility automatically.
- [x] `--claude` proves launch, route selection, subagent, workflow, hook, and
  trace behavior using the installed Claude CLI.
- [x] `--include-anthropic` proves first-party routing and switching to every
  configured non-blocked, tool-compatible alias and back in the same launch;
  tool-disabled aliases are proven through explicit launches.
- [x] `ccr doctor` is offline by default; `--live` probes one alias per provider
  and `--live --all` probes every non-blocked alias with bounded concurrency.
- [x] Human and `schema_version: 1` JSON results agree and return nonzero when a
  required check fails.

## Team Profiles

- [x] Export is deterministic and includes only provider/model configuration,
  capabilities, compatibility, and non-secret credential requirements.
- [x] Environment variable names may be exported, while raw values, keychain
  identifiers, and credential-file paths are never exported.
- [x] Import rejects unknown fields, invalid values, unsupported versions, and
  oversized manifests before writing.
- [x] Conflict mode defaults to atomic failure. Dry-run performs the identical
  validation and planning path without database changes.
- [x] Export followed by import into an empty database preserves routing
  configuration and requires local credential binding where necessary.

## Compatibility And Regression

- [x] Schema v3 migrates transactionally to v4 while preserving providers,
  models, legacy launch records, agents, and conformance history.
- [x] Registered aliases remain visible in `/model`, including selective
  escaping of Claude family substrings and legacy gateway IDs.
- [x] Fixture E2E covers Anthropic-compatible and OpenAI-compatible providers,
  full/degraded/chat-only modes, malformed responses, cancellation, model
  switching, agents, workflows, hooks, status line, and traces.
- [x] Fixture E2E runs without skips on pull requests, `main`, and daily against
  pinned Claude Code 2.1.209 and the current npm release.
- [x] The local real matrix passes against first-party Anthropic and every
  configured non-blocked alias, currently LiteLLM `glm-5-2`.

## Quality And Release

- [x] `go test -race -count=1 -p 4 ./...`
- [x] `go vet ./...`
- [x] `golangci-lint run ./...`
- [x] `govulncheck ./...`
- [x] Required fixture live target passes without skips.
- [x] Required local real-provider matrix passes without skips.
- [x] The required `codex review ... --uncommitted` command reports no findings.
- [ ] Pull-request CI is green and the PR body contains concise sections for

  release notes.

  formula updates.
