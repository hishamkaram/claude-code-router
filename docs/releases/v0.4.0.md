# CCR v0.4.0

## Highlights

- Local Claude subscription accounts can be registered once and selected as a
  pool for interactive Claude Code sessions.
- Plain interactive pool launches automatically move to the next usable
  account after a first-party Anthropic HTTP 429 by stopping Claude Code and
  relaunching with `--continue`.
- Account credentials remain in the OS keychain. SQLite and CLI output contain
  redacted metadata and keyring references only.

## User-Visible Changes

Register and inspect local accounts with the new command group:

```bash
ccr claude-account import personal --from current
claude setup-token
ccr claude-account import work --oauth-token-stdin
ccr claude-account list
ccr claude-account show personal
ccr claude-account test personal
ccr claude-account refresh personal --from current
ccr claude-account disable work
ccr claude-account enable work
ccr claude-account remove work --yes
```

Start an automatically selected or explicitly selected account:

```bash
ccr launch --auth-mode subscription-pool
ccr launch --auth-mode subscription-pool --claude-account personal
```

- Automatic selection uses the least recently selected enabled, unexpired
  account whose cooldown has elapsed.
- The selected account is visible in launch output and `ccr status`.
- Stable, schema-versioned JSON is available from `claude-account list --json`
  and `claude-account show <name> --json`.
- Invalid account commands and incompatible launch options fail before launch
  or persistence side effects.

## Rotation Boundaries

- Account identity remains fixed for each Claude Code process. CCR never swaps
  OAuth identity inside a running process.
- Automatic relaunch is limited to a plain interactive pool launch without
  `--print`, an explicit account, managed CUA options, or extra Claude Code
  arguments.
- Explicit-account and unsupported launch shapes report exhaustion visibly and
  require a new launch.
- Provider-route 429 responses do not trigger subscription-account rotation.
- When no usable account remains, CCR fails without falling back to the default
  Claude login, an Anthropic API key, or another provider.

## Privacy and Security

- OAuth access and refresh tokens are stored only in the OS keychain.
- SQLite stores account metadata and dedicated redacted keyring references,
  never raw OAuth tokens.
- Import fails before SQLite persistence when the OS keychain is unavailable.
- Removal deletes keychain credentials before account metadata; failed cleanup
  leaves metadata available for a safe retry.
- Account names and state are observable, while token values and keyring
  locations are not printed.

## Platform Notes

- On Linux and Windows, `--from current` reads the current Claude credentials
  file when it exists with safe permissions.
- On macOS, use `claude setup-token` and pass the generated token through
  `--oauth-token-stdin`; CCR does not extract Claude's active Keychain login.
- Subscription pools are intended for local individual use. Teams, automation,
  hosted products, and third-party integrations should use official Anthropic
  API authentication.

## Upgrade

```bash
brew upgrade hishamkaram/tap/claude-code-router
ccr version
ccr doctor
ccr claude-account list
```

## Verification

Release evidence is recorded in `docs/acceptance/v0.4.0.md`. Required coverage
includes deterministic Go gates, real Claude Code process-relaunch proof, the
12-job Linux/macOS fixture matrix across pinned/latest Claude Code and all
three gateway protocols, the configured real-provider matrix, release archive
checksums and provenance, and signed GHCR browser-image verification.

## Known Limits

- Pool selection is load balancing, not an exclusive process-lifetime lease;
  overlapping launches may reuse an account after all eligible accounts have
  been selected.
- The macOS computer-use helper remains a source-built unsigned preview. It is
  not included in Homebrew or GoReleaser archives.
